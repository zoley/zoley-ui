# rengar-admin 项目总览与架构拆解

## 一句话理解这个项目

`rengar-admin` 是一个基于 `Vue 3 + Vite + TypeScript` 的后台管理系统模板。它想解决的不是“做一个花哨的演示站”，而是“给你一套干净、轻量、可扩展的后台项目底盘”。

你可以把它理解成一栋已经搭好钢结构的大楼：

- `src/views` 是每个业务房间
- `src/layouts` 是整栋楼的骨架
- `src/router` 是楼层导视系统
- `src/stores` 是共享控制中心
- `src/api` 是对外通讯系统
- `build` 和 `packages` 是施工设备和定制工具

## 技术栈分别负责什么

### 1. Vue 3

负责组件化开发。

你看到的大部分 `.vue` 文件，都是 Vue 单文件组件。这个项目使用的是 `script setup` 风格，这是 Vue 3 里最主流的写法之一，代码更短，也更贴近组合式 API 的思维。

### 2. Vite

负责开发服务器、打包构建、插件体系。

这个项目很多“自动能力”都不是 Vue 本身提供的，而是 Vite 插件提供的，比如：

- 自动路由生成
- 自动导入 API
- 自动导入组件
- UnoCSS 支持
- 图标能力

### 3. TypeScript

负责类型约束和编辑器提示。

对初学者来说，TypeScript 一开始会让代码显得更复杂，但长期收益很大，因为它能让项目结构更稳定，减少“参数乱传”“返回值不确定”“路由名写错”这类问题。

### 4. Vue Router

负责页面切换和路由守卫。

它不只是“切页面”，还承担：

- 访问控制
- 菜单生成的数据来源
- 标签页与 keep-alive 的基础数据

### 5. Pinia

负责跨组件共享状态。

在这个项目里，Pinia 不是拿来堆一切数据的，而是主要承载“全局性状态”：

- 用户登录状态
- 权限数据
- 菜单数据
- 标签页数据
- 布局与主题配置

### 6. Naive UI

负责 UI 组件能力。

比如消息提示、下拉菜单、按钮、抽屉、表单等，都可以直接基于它来搭建，不需要重复造轮子。

### 7. UnoCSS

负责原子化样式。

你会看到很多类似 `flex items-center gap-4 px-4` 这样的类名。这套写法接近 Tailwind，但这里用的是 UnoCSS。好处是：

- 写样式速度快
- 结构和样式距离近
- 更适合后台系统这种大量“布局型界面”

## 目录结构怎么理解

下面是你最需要记住的目录分层。

### `src/`

业务主目录。

- `main.ts`：应用入口
- `App.vue`：根组件，负责选择布局
- `views/`：业务页面
- `layouts/`：后台框架骨架
- `router/`：路由配置与守卫
- `stores/`：Pinia 状态
- `api/`：接口调用层
- `hooks/`：组合式逻辑封装
- `components/`：通用组件
- `utils/`：工具函数
- `config/`：应用级配置

### `build/`

Vite 构建期相关能力。

这里不是“运行时业务代码”，而是“项目编译时的辅助逻辑”，例如：

- 注册哪些 Vite 插件
- 路由插件怎么接入
- 代理怎么创建

### `packages/`

项目内自维护的小型基础包。

这是这个项目很值得你学的点。作者没有把所有工具函数都塞到 `src/utils`，而是把一些具备复用价值的能力拆成 package，比如：

- `axios` 基础请求类
- `color` 主题色工具
- `unocss` 配置能力
- `vite-plugin-vue-routes` 自动路由插件

这是一种很典型的“把通用能力往外抽”的工程思路。

### `typings/`

类型声明目录。

这里通常放自动生成或手写的全局类型，例如自动导入的类型声明、路由名类型等。

## 这套项目的核心设计思想

## 1. 轻量而不是臃肿

很多后台模板喜欢一开始塞进非常多功能，导致你改一个地方就牵一大片。这个项目明显在克制功能范围，只做后台通用底座。

这说明作者在追求：

- 好上手
- 好修改
- 好扩展

这是很好的工程价值观。

## 2. 约定优于配置

比如页面放在 `src/views`，然后通过插件自动生成路由。这种方式的重点不是“少写一点配置”，而是“降低多人协作时的认知成本”。

大家都知道页面该放哪、路由大概会长什么样，协作效率会高很多。

## 3. 分层明确

你会发现这个项目很少出现“页面里又写权限、又写请求、又写全局状态、又写布局控制”的混乱情况。它倾向于：

- 页面只关心页面
- store 只关心共享状态
- router 只关心路由和守卫
- request 只关心请求流程

这就是工程项目可维护的关键。

## 4. 在通用性和简单性之间找平衡

它没有做成超重型平台，也没有简单到什么都没有，而是做了“刚刚够用”的抽象。

这是很成熟的一种工程判断：

- 太少，会导致业务重复造轮子
- 太多，会导致模板本身难以理解和修改

## 你应该重点吸收什么

如果你现在是初学者，我建议你把这个项目当成“后台项目组织方式”的样板来学，而不只是 Vue 语法练习。

你最应该学会的是：

1. 一个中后台项目应该怎样分目录
2. 页面、路由、布局、状态、请求应该怎样分层
3. 什么逻辑适合写到组件里，什么逻辑适合抽到 store / hook / utils / package
4. 如何把“常用能力”做成通用机制，而不是在每个页面复制粘贴

这些能力，比单独记住某个 Vue API 更重要。

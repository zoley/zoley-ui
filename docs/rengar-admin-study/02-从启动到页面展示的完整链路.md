# 从启动到页面展示的完整链路

这一篇非常重要。你只要把应用启动链路看懂了，后面看任何模块都会轻松很多。

## 第 1 步：浏览器加载入口文件

入口文件是 `src/main.ts`。

这个文件的职责非常纯粹：创建应用、注册能力、等待路由准备完成、挂载到页面。

核心流程是：

1. 引入基础样式和 UnoCSS
2. 创建一个加载中的应用 `AppLoading`
3. 创建真正的主应用 `AppProvider`
4. 注册 Pinia
5. 注册插件
6. 注册并等待路由
7. 挂载主应用
8. 卸载加载动画应用

这种写法有个很实用的巧思：

- 加载动画和主应用拆成两个小应用
- 主应用在路由准备好之后再挂载
- 用户感知会更平滑

这比一开始就直接渲染空白页面更友好。

## 第 2 步：根组件决定当前使用哪种布局

根组件是 `src/App.vue`。

它并没有直接渲染某个具体页面，而是先判断当前路由要使用哪一种布局：

- 默认用 `BasicLayout`
- 如果路由 `meta.layout` 指定为 `blank`，就用 `BlankLayout`

这是一种很常见的后台项目技巧。

为什么要这样做？

- 登录页、404 页通常不需要侧边栏和顶部栏
- 后台页面通常需要完整框架布局
- 用“布局层”统一包住页面，比每个页面自己判断更整洁

所以页面并不是直接长在根组件下面，而是：

`App.vue -> Layout -> RouterView -> 具体页面`

## 第 3 步：Pinia 建立全局状态中心

`src/stores/index.ts` 里通过 `createPinia()` 创建状态容器，并挂上 `pinia-plugin-persistedstate`。

这意味着项目支持把部分 store 状态持久化到浏览器存储中。

它的价值非常直观：

- 刷新页面后，某些状态还能保留
- 标签页历史、搜索历史之类的数据不必重新生成

这就是典型的“基础设施先搭好，业务使用更轻松”。

## 第 4 步：注册插件

`src/plugins/index.ts` 当前主要做的是注册自定义指令。

这一层的作用是把“全局通用能力”集中初始化，而不是散落在某个组件里。项目规模一大，这种集中式初始化非常有帮助。

## 第 5 步：创建路由实例

`src/router/index.ts` 做了几件事：

1. 使用 `createWebHistory(import.meta.env.VITE_BASE_URL)` 创建 history
2. 合并自动生成路由和额外配置
3. 补上根路径重定向和 404 兜底路由
4. 在 `setupRouter` 中注册全局守卫并等待 `router.isReady()`

这里的重点是“路由不是纯手写的”，而是自动路由和额外配置共同作用的结果。

## 第 6 步：自动路由机制开始发挥作用

路由数据来源主要是 `src/router/routes.ts`，但这个文件不是手写主导，而是由 `vite-plugin-vue-routes` 自动生成。

生成逻辑来自：

- `build/plugins/router.ts`
- `packages/vite-plugin-vue-routes/`

这套机制的核心思想是：

- 你在 `src/views` 下按目录组织页面
- 构建期扫描这些页面
- 自动输出标准的 `RouteRecordRaw[]`

这样做的好处：

- 少写重复路由配置
- 目录结构和路由结构更一致
- 更适合后台项目的大量页面管理

这其实是很多团队都会采用的一类思路，只是实现方式各不相同。

## 第 7 步：全局路由守卫决定能不能进入页面

`src/router/guard.ts` 是权限流转的关键节点。

它会依次处理这些情况：

1. 常量路由直接放行
2. 如果访问登录页，已登录就跳首页
3. 如果未登录且访问受保护页面，就跳登录页
4. 如果登录了但用户详情没拿到，就先请求用户信息
5. 用户信息回来后，再生成菜单、初始化标签页、初始化 keep-alive 数据
6. 如果目标路由要求权限，就校验权限
7. 没权限就跳到 404

你要注意一个非常好的工程点：

“登录态建立”和“用户详情加载完成”被拆成了两个阶段。

很多新手项目常见写法是：

- 只要本地有 token，就认为一切准备好了

但这个项目更严谨：

- 有 token 只是“可能已登录”
- 用户详情、权限码、菜单初始化完成后，才算真正完成权限上下文建立

这是更接近真实后台系统的设计。

## 第 8 步：页面最终被渲染出来

当路由、布局、权限都准备好了，页面才会真正显示到 `RouterView` 中。

所以你看到一个页面出现，背后往往已经经历了这条链路：

`main.ts -> App.vue -> layout -> router -> guard -> stores -> page`

## 这套链路为什么值得学

因为它把“用户看到页面之前必须完成的前置工作”组织得很清楚：

- 样式准备
- 状态准备
- 路由准备
- 权限准备
- 布局准备

初学者常见问题是把这些逻辑都堆进页面里，结果页面又长又乱。这个项目给你的启发是：

“页面展示之前的公共流程，尽量前置到入口、路由守卫、全局状态中处理。”

## 你可以自己做的练习

1. 在 `main.ts` 中给每一步加 `console.log`
   看看启动顺序是不是和这篇文章说的一样。

2. 在 `guard.ts` 里断点
   看未登录、已登录、无权限时分别走哪条分支。

3. 给一个页面手动加 `meta.layout = 'blank'`
   看它是不是就切换成空白布局了。

如果你把这三件事做完，整个项目的主流程就算真正入门了。

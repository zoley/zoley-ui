# 路由、权限、状态管理与工程巧思

这一篇会进入这个项目最有学习价值的部分：为什么它的菜单、权限、标签页、缓存、主题可以互相配合，而且代码没有乱成一团。

## 一、路由不只是页面跳转

很多初学者理解路由时，只把它当成“地址栏切换页面”的工具。但在后台项目里，路由往往还是下面这些能力的中心数据源：

- 菜单生成
- 标签页生成
- 权限判断
- 面包屑生成
- 页面缓存规则
- 布局切换

这个项目就是典型例子。

在 `src/router/routes.ts` 里，每个路由除了 `path`、`component` 之外，还有很多 `meta` 信息，例如：

- `title`
- `icon`
- `order`
- `roles`
- `hideInMenu`
- `hideInTab`
- `keepAlive`
- `layout`
- `fixedInTab`

这说明一个页面真正的“身份信息”，不是散落在多个地方，而是尽量统一挂在路由元信息里。

这是行业里非常常见的后台项目设计方式。

## 二、为什么自动路由依然允许手动修正

自动路由很方便，但纯自动也容易不灵活。所以这个项目用了“自动生成 + 人工补充”的组合方式。

关键代码在 `src/router/index.ts` 和 `src/router/utils.ts`：

- 自动生成的路由来自 `routes`
- 人工额外配置来自 `extraRoutesConfig`
- 两者通过 `mergeRouteConfig` 合并

这样做的好处是：

- 页面结构可以跟着目录自动长出来
- 但菜单图标、排序、额外行为仍然可控

这是一个很成熟的折中方案。

## 三、权限控制为什么放在路由守卫，而不是每个页面里判断

`src/router/guard.ts` 会在进入页面前统一做权限检查。

这样做有几个核心好处：

### 1. 入口统一

如果每个页面自己判断权限，你会得到很多重复代码，而且容易漏掉某些页面。

统一放到路由守卫后：

- 所有页面进入前都会经过这里
- 逻辑集中，便于修改
- 更容易定位问题

### 2. 用户体验更完整

它不只是拦住页面，还会顺带完成：

- 拉取用户详情
- 初始化菜单
- 初始化标签页
- 初始化 keep-alive

也就是说，它处理的是“进入系统之前的上下文准备”。

### 3. 更符合后台系统的思维

后台项目里“能不能看某个页面”本质上就是路由访问控制问题，把它放到路由层最自然。

## 四、Pinia 的几个 store 是怎么分工的

这个项目的 store 拆分很值得学。

## 1. `app` store

负责“应用级配置与 UI 状态”。

例如：

- 当前布局模式
- 是否折叠菜单
- 是否显示配置抽屉
- 是否显示搜索弹窗
- 主题模式
- 是否灰度模式/色弱模式

这类状态的共同点是：

- 不属于某个单页
- 影响多个组件
- 需要全局共享

所以放到 `app` store 很合理。

其中最值得学的是主题切换设计。

它不只是切一个布尔值，而是把主题模式分成：

- `light`
- `dark`
- `auto`

再结合系统主题、CSS 类名、Naive UI 主题覆盖、视图过渡动画一起工作。这是比较完整的主题系统设计。

## 2. `auth` store

负责用户与权限基础数据。

包含：

- token
- 用户 id
- 用户名
- 权限码 `roleMap`

这里有个巧思：

权限不是存成数组后每次遍历，而是存成 `Map`。这样做权限判断时可以直接 `has()`，语义更清楚，也更适合做频繁查询。

## 3. `menu` store

负责菜单树和搜索历史。

它通过过滤路由生成菜单，而不是额外维护一套菜单配置。这种做法的好处是“单一数据源”。

也就是：

- 路由定义页面
- 菜单从路由派生

这样不容易出现“页面有了，菜单没配”或者“菜单配了，但路由忘了写”的双份维护问题。

## 4. `tab` store

负责多标签页能力。

它会监听当前路由变化，自动把当前页面加入标签列表。并且支持：

- 关闭当前
- 关闭其他
- 关闭左侧
- 关闭右侧
- 关闭全部

这类功能是典型后台系统高频交互，把它独立成 store 很合理。

## 5. `keepAlive` store

负责页面缓存名单。

它不是简单地“所有页面都缓存”，而是根据路由配置和当前可见标签页联合决定真正的缓存列表。

这个思路很实用：

- 想缓存的页面要先通过路由 `meta` 声明
- 但如果标签页已经关闭，也没必要继续缓存

这能减少无意义缓存，兼顾体验和资源使用。

## 五、菜单、标签页、缓存为什么能协同工作

因为它们都围绕“路由”展开。

你可以把它理解成：

- 路由是事实来源
- 菜单是路由的导航投影
- 标签页是路由的访问记录投影
- keep-alive 是路由的缓存策略投影

这是很重要的工程思维：多个功能不要各自维护一份相似数据，而应该围绕一个稳定核心做派生。

## 六、请求层为什么分成基础包和业务包两层

`packages/axios/index.ts` 提供的是“基础 HTTP 客户端能力”，例如：

- 创建 axios 实例
- 注入拦截器钩子
- 管理 `AbortController`
- 提供 `get/post/request/upload`
- 取消所有请求

而 `src/api/request.ts` 负责“项目级业务定制”：

- 自动带上 token
- 记录当前路由路径
- 统一处理后端响应码
- 401 时清空登录态并跳转登录

这种设计很专业，因为它把：

- 通用能力
- 项目策略

分开了。

以后如果你换一个项目，只要后端协议不同，可能主要改的是 `src/api/request.ts`，而不是把整个基础请求库推倒重来。

## 七、为什么 401 处理里要取消所有请求

这是一个很真实的工程细节。

如果某个请求返回 401，说明登录已失效。此时如果系统里还有其他接口同时在飞：

- 它们可能继续报错
- 可能反复弹错误提示
- 可能重复触发跳转

所以这里调用 `cancelAll()`，本质上是在做“失效状态熔断”。

这类处理非常像真实生产项目中的鲁棒性设计，而不是演示代码。

## 八、为什么布局层要拆成组件而不是一个超大文件

`src/layouts/components/` 里把面包屑、标签页、搜索框、页脚、配置抽屉等拆成很多小组件。

这有两个直接收益：

1. 每个文件职责单一，更容易读
2. 布局骨架可调整，可替换，可复用

这也符合前面说的工程原则：一个文件太大，通常意味着职责过多。

## 九、这个项目体现了哪些行业通用方法

### 1. 单一数据源

围绕路由元信息派生菜单、标签页、缓存规则。

### 2. 前置统一拦截

在路由守卫做登录与权限控制，在请求拦截器做 token 注入和异常处理。

### 3. 组合式封装

用 `hooks`、`store`、`utils` 抽离页面外逻辑，避免单组件膨胀。

### 4. 构建期自动化

把自动路由、自动导入、图标处理放到构建层，而不是运行时硬写死。

### 5. 适度抽象

只抽真正有复用价值的能力，不做为了抽象而抽象的复杂设计。

## 十、你应该怎样把这些思想迁移到自己的项目

1. 不要一开始就在页面里堆所有逻辑
2. 先找“全局共享的数据”和“页面局部的数据”边界
3. 能从路由派生的信息，就尽量不要再维护第二份
4. 重复出现两次以上的流程，就考虑抽成 hook、utils 或基础包
5. 只有在“能明显降低维护成本”时才抽象

如果你能把这 5 条慢慢变成习惯，你的项目会越来越像真正可维护的工程，而不是一堆能跑的页面。
